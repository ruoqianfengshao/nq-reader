# NQ Reader

NQ Reader 是一个纯前端的 NodeQuality 报告阅读器。粘贴 NodeQuality 生成的 Markdown 报告后，页面会提取硬件、IP 与网络指标，将原始测试结果转换为更容易理解的用途结论。

在线地址：[nq-reader.cc.cd](https://nq-reader.cc.cd/)

## 功能

- 解析 NodeQuality Markdown 报告，不上传报告内容
- 汇总 CPU、内存、磁盘、IP、流媒体和网络数据
- 分别判断代理、AI / 流媒体、建站和存储用途
- 展示结论对应的原始指标与档位
- 生成适合分享的结论海报，并支持复制为图片
- 提供浅色、深色主题和内置示例报告
- 首页底部展示 NodeSeek 延伸阅读，导航资料按钮在全站可用
- 结果页宽屏固定显示参考资料，窄屏和移动端通过导航资料按钮打开

当前不支持直接读取报告 URL。使用时需要复制 NodeQuality 报告正文并粘贴到输入框。

## 本地运行

环境要求：Node.js 18 或更高版本。

```bash
npm install
npm run dev
```

生产构建：

```bash
npm run build
```

构建结果位于 `dist/`。

## 结论模型

解析过程分为三步：

1. 从报告的硬件、IP 和网络章节提取结构化证据。
2. 将每条证据归为 `良好`、`一般` 或 `较差`，并保留原始数值和档位。
3. 按用途组合证据，生成代理、AI / 流媒体、建站和存储结论。

结论不是 NodeQuality 官方评分，也不是购买建议。价格、商家信誉、测试时间、晚高峰表现和实际用户地区仍需单独判断。

### 基础指标档位

以下是当前主要硬件阈值。完整实现以 `src/analyzer.ts` 为准。

| 指标 | 较差 | 一般 | 良好 |
| --- | --- | --- | --- |
| Sysbench 单核 | `< 800` | `800–1199` | `>= 1200` |
| 内存容量 | `< 768 MB` | `768–2047 MB` | `>= 2048 MB` |
| 内存读取 | - | `< 20000 MB/s` | `>= 20000 MB/s` |
| 内存延迟 | `>= 250 ns` | `180–249 ns` | `< 180 ns` |
| 硬盘容量 | `< 20 GB` | `20–39 GB` | `>= 40 GB` |
| 4K 随机读写 | `< 20 MB/s` | `20–39 MB/s` | `>= 40 MB/s` |
| 4K IOPS | `< 5000` | `5000–19999` | `>= 20000` |
| 顺序读写 | - | `< 300 MB/s` | `>= 300 MB/s` |

部分良好区间还会细分为“不错”“很好”“很强”“顶级”，用于结论明细展示，不改变三档状态颜色。

### 代理结论

代理用途综合以下证据：

- 电信、联通、移动的综合评级
- 各地区三网大包延迟与丢包
- CN2、CMIN2、AS9929、AS10099 等回程线路
- 国内测速错误、国际互连错误与重传
- 回程绕路情况

判断优先级：

1. 三网方向都明确使用 CN2GIA/CTGGIA，且没有主干混合或绕路时，给出“三网 CN2GIA”。
2. 电信为 CN2GIA、联通为 AS9929（包括 AS10099 → AS9929）、移动为 CMIN2，且三条主干没有混合或绕路时，给出“三网顶级线路”。
3. 三网均有优质回程但未达到上述顶级模式时，给出“三网精品线路”或具体运营商线路。
4. 没有明确线路标签时，优先找各运营商表现快乐的地区。
5. 三网综合表现都优秀时可判为三网快乐。
6. 部分运营商一般时标记“看地区”。
7. 仅有较差运营商或明显网络异常时判为备用机，不建议作为对应方向的主力线路。

低延迟本身不会直接等同于精品线路，仍需结合回程和测速结果。

线路标签按运营商分别识别：电信的 CN2 GIA、联通的 AS9929/AS10099 或借道优质线路、移动的 CMIN2 可共同归纳为“三网精品线”，不会把三网统一标记为“三网 CN2GIA”。

联通方向单独出现 AS10099 只算精品；明确出现 AS9929 或 AS10099 → AS9929，且主干没有混合或绕路时，可进入顶级线路判断。AS9929 之后在国内省网末端接入 AS4837/CU169 不单独降低主干线路等级。

线路等级与实际体验分开判断。测速 ERROR、接收为 0、丢包或国际互连异常不会改变已确认的主干线路等级，但会增加“部分方向异常”标签。

### AI / 流媒体结论

该结论综合服务解锁、IP 使用地、送中风险、黑名单和 IP 风险：

- 服务状态为“解锁”记为良好，“仅 APP”记为受限，其余记为不可用。
- 出现明显送中风险、至少 3 个服务不可用、黑名单命中，或至少 2 个 IP 风险项较高时，判为流媒体风控机。
- 至少 4 个服务解锁且 ChatGPT 可用时，判为 AI / 流媒体快乐机。
- 至少 3 个服务解锁时，判为流媒体可用。
- 仅少量服务解锁时，判为部分流媒体可用。

#### 落地机

同时满足以下条件时判为落地机：

- IP 使用地明确且不在中国大陆
- 至少 3 个服务解锁
- 最多 1 个服务不可用
- 没有明显送中风险
- 没有检测到黑名单风险；报告缺失该项时不会单独阻断判断
- 最多 1 个高风险 IP 项

结论会尽量带上使用地区，例如“香港落地机”“美国落地机”。

### 建站结论

建站用途读取 CPU 单核、内存容量和吞吐、内存延迟、4K 随机读写及 IOPS。

- 所有可用证据均良好时，判为轻量站可用。
- 出现一般指标时，判为备用小站。
- 任一核心指标较差时，判为不适合数据库站。

这是偏保守的组合规则。静态站和低流量服务可能不受单项短板明显影响，但 WordPress、MySQL 和多容器服务应重点关注内存与 4K 性能。

### 存储结论

存储用途读取磁盘容量、顺序读写和 4K 随机读写：

- 大容量磁盘且顺序读写良好、4K 没有明显短板时，判为大盘机。
- 大容量磁盘但 4K 较差时，标记“大盘机，数据库谨慎”。
- 非大盘但顺序读写良好时，判为轻量下载可用。
- 4K 随机性能较差时，判为不适合存储服务。
- 其余情况判为备用盘。

当前“大容量”通过报告中的 TB 容量或三位数以上 GB 容量识别。

## 首页标签规则

首页每个示例最多展示两个标签，选择顺序如下：

1. 汇总代理、AI / 流媒体、建站、存储四个结论产生的全部标签。
2. 按 `良好 → 一般 → 较差` 排序。
3. 同一等级保持结论顺序：代理、AI / 流媒体、建站、存储。
4. 同一结论内保持标签生成顺序。
5. 选出第一个良好标签后，如果后续结论也有良好标签，第二个优先取下一结论的最佳标签。
6. 只有其他结论没有良好标签时，才取同一结论的第二个标签。

代理的“顶级”和“精品”在标签中明确显示为“顶级线路”和“精品线路”，避免被误解为整台机器的综合等级。

代理运营商标签按最终综合档位互斥：顶级显示为“电信/联通/移动优化”，快乐显示为“电信/联通/移动快乐”，同一运营商不会同时出现“优化”和“快乐”。三网同档时合并为“三网优化”或“三网快乐”。

代理结论达到精品线路、顶级线路、三网 CN2GIA，或至少一个运营商达到快乐时，增加“线路鸡”归类标签。

代理分享海报按“线路鸡 → 具体线路等级 → 运营商快乐/异常提示”的顺序展示，并隐藏重复的运营商精品线路标签，最多展示三个标签。存在“部分方向异常”时，该提示优先占第三位。快乐和线路鸡标签统一使用良好状态颜色。

## 代码结构

```text
src/
├── analyzer.ts          # 报告拆分、证据提取和基础指标档位
├── analyzerTypes.ts     # 分析结果与证据类型
├── rules/useCases.ts    # 四类用途结论规则
├── examples.ts          # 首页示例报告
├── resources.ts         # 参考资料来源、站点图标与链接
├── App.tsx              # 结果展示、标签和分享海报逻辑
└── styles.css           # 浅色/深色界面样式
```

修改结论规则时，应同步更新本 README，并使用多份不同地区、线路和配置的 NodeQuality 报告进行回归验证。
